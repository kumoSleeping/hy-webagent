# pi-web-platform 安全修复方案

> 本文档只描述修复方向与方案，不包含实际代码改动，由开发者按此执行。
> 配套的风险清单见画布 `security-audit`。共 9 项，按优先级组织如下。

## 修复优先级总览

| 阶段 | 目标 | 涉及项 |
|------|------|--------|
| 第一阶段（上线前必须） | 切断「普通用户 → 管理员主密钥/厂商凭证」的完整接管链 | F1、F2 |
| 第二阶段（尽快） | 收敛对外攻击面：SSRF、暴力破解、DoS、跨站握手 | F3、F4、F5 |
| 第三阶段（加固） | 消除虚假安全感、补齐基础加固 | F6、F7、F8、F9 |

核心判断：**这套系统真正的信任边界应该是「操作系统 / 容器级隔离」，而不是提示词或字符串黑名单。** 所有修复都应围绕这个原则展开——把「防御」从模型自觉（软性提示）迁移到进程无法逾越的硬边界（文件系统、网络、权限）。

---

## 第一阶段：切断接管链

### F1 — 给 Agent 的 shell / 文件工具建立真正的隔离边界（严重）

**问题本质**：PI Agent 以 `workspaces/<user>/projects` 为工作目录运行，但它拥有 `bash` 和文件读写工具，且进程运行在宿主上，没有任何 chroot / 沙箱。`isolation.ts` 的路径校验只挡住了 HTTP 的 `/api/files/*` 接口，管不到 Agent 自己发起的工具调用。因此一句 `cat ../../../data/users.json` 就能越权。

**修复方向（按隔离强度从高到低，建议优先选前者）**：

1. **每用户独立容器 / 沙箱进程（首选）**
   - 每个用户会话在独立的容器或受限子进程中运行 Agent，容器内只挂载该用户自己的 workspace（读写），其余宿主路径一律不可见。
   - 关键点：`data/`（用户库、管理员密钥）、`~/.pi/agent`（厂商凭证）、其他用户的 workspace **绝不能出现在 Agent 进程可达的路径树里**。当前 compose 把 `~/.pi/agent` 只读挂载进容器，等于把厂商凭证送到 Agent 手边，需要重新设计凭证下发方式（例如由平台层代理调用模型、Agent 侧不持有原始 key）。
   - 用非 root 用户运行，配合只读根文件系统、`--cap-drop=ALL`、seccomp、内存/CPU/进程数限制，防止 F1 顺带被用于 DoS 或提权。

2. **文件系统层强制 chroot / 挂载命名空间**
   - 若不便上容器，至少让 Agent 进程 chroot 到用户目录，或用 mount namespace 只暴露该用户 workspace，使 `../` 在物理上无法逃逸。

3. **工具层白名单（最低限度，作为纵深防御而非唯一防线）**
   - 把已定义却从未接入的 `checkDangerousCommand()` 真正挂到命令执行链路上，并从「黑名单」改为「命令 + 参数白名单」。
   - 对文件工具的路径同样复用 `validatePath` 式的校验，把所有落盘/读取都收敛到用户目录内。

**验收标准**：在 Agent 会话里尝试读取 `data/admin-key.txt`、`data/users.json`、`~/.pi/agent/auth.json`、其他用户 workspace，全部失败。

---

### F2 — 管理员主密钥：不落盘明文、拒绝弱默认、恒定时间比较（严重）

**问题本质**：启动时把管理员密钥明文写入 `data/admin-key.txt`；`config.adminKey` 默认硬编码 `sk-hyw-DevKey9ChangeMe24`，compose 默认 `change-me-in-production`。该主密钥能直接换取完整管理员权限。

**修复方向**：

1. **不再落盘明文密钥**
   - 首次启动生成的管理员 key 只打印到控制台一次，提示运维立即保存；不写入任何文件。
   - 若确实需要引导态可读，改为存哈希，并在首次登录后强制轮换。

2. **拒绝弱 / 默认密钥启动**
   - 启动时若 `ADMIN_KEY` 缺失或等于已知占位值（`change-me-in-production`、`sk-hyw-DevKey9ChangeMe24` 等），直接拒绝启动并给出明确报错，杜绝「忘了改」。
   - 主密钥只经环境变量注入，不在代码里保留任何可用默认值。

3. **恒定时间比较**
   - `adminAuthMiddleware` 里 `bearer === config.adminKey` 改用 `crypto.timingSafeEqual`，避免时序侧信道。

4. **清理现存泄漏**
   - 删除现有 `data/admin-key.txt`，轮换当前管理员 key（因为它已明文落盘且可能已被 F1 读走）。

**验收标准**：使用默认/占位密钥无法启动；正常启动后磁盘上不存在明文主密钥文件。

## 第二阶段：收敛对外攻击面

### F3 — 修复 `/api/remote/fetch` 与 `/api/remote/download` 的 SSRF（高）

**问题本质**：已登录用户可让服务器 `fetch()` 任意 URL，无 scheme / host 校验、不拦内网 IP，且 `/remote/fetch` 把响应体回传客户端，构成可读 SSRF。

**修复方向**：

1. **协议白名单**：仅允许 `http` / `https`，拒绝 `file:`、`gopher:`、`ftp:` 等。
2. **目标地址校验（关键）**：解析主机名 → 解析出的 IP 若落在私有/保留网段（`10/8`、`172.16/12`、`192.168/16`、`127/8`、`169.254/16`、`::1`、`fc00::/7` 等）一律拒绝。注意要对**解析后的实际 IP**判断，防 DNS rebinding。
3. **禁止跟随跳转到内网**：手动处理 redirect，对每一跳都重新做上面的 IP 校验，或直接禁用自动重定向。
4. **限额**：保留现有大小与超时限制，并对返回内容类型做约束。
5. **可选加固**：出站统一走带白名单的正向代理，从网络层兜底。

**验收标准**：请求 `http://169.254.169.254/...`、`http://127.0.0.1:...`、`http://<内网IP>` 均被拒绝。

---

### F4 — 限流 + 高效 API key 校验（高）

**问题本质**：`findUserByApiKey` 对每个用户逐一 `bcrypt.compare`（rounds=12），登录和每个 admin 请求都触发一轮全表 bcrypt，既可暴力枚举又是 CPU DoS 放大器；`config` 里的 `rateLimit*` 定义了却从未接入。

**修复方向**：

1. **接入限流**：用 `express-rate-limit`（或等价方案）落地 `config` 里已有的 `rateLimitWindowMs` / `rateLimitMaxRequests`，对 `/api/auth/login` 单独设更严格的阈值。
2. **改造 key 校验，避免全表 bcrypt**：
   - 给 API key 加不可枚举的**查找索引**（例如对 key 取 HMAC/SHA-256 得到定长指纹，存在用户记录上并建立映射），登录时先按指纹定位唯一用户，再对该用户做**单次** bcrypt 验证。
   - 这样每次校验只跑一次 bcrypt，既防枚举又消除 DoS 放大。
3. **失败退避**：对连续失败登录做指数退避或临时锁定。

**验收标准**：无论用户量多少，单次登录/鉴权只执行一次 bcrypt；高频登录请求被限流拦截。

---

### F5 — WebSocket 升级校验 Origin（中）

**问题本质**：`server.on("upgrade")` 只校验 query 里的 `sessionId`，不校验 `Origin`，HTTP 的 CORS 管不到 WebSocket 握手，存在跨站 WebSocket 劫持（CSWSH）面。

**修复方向**：

- 在 upgrade 阶段读取 `Origin` 头，与允许列表（复用 `config.corsOrigin`）比对，不匹配则 `socket.destroy()`。
- 同时避免把 `sessionId` 放在 URL query（易进日志/被引用），条件允许改用握手期的鉴权消息或 Cookie + CSRF 令牌。

**验收标准**：来自非白名单 Origin 的 WebSocket 握手被拒绝。

## 第三阶段：加固与消除虚假安全感

### F6 — 重新定位「敏感文件过滤」与「提示注入检测」（中）

**问题本质**：`checkSensitive` 只按 basename 子串匹配少数名字，漏掉 `.ssh/config`、`authorized_keys`、`.git/config`、`.bash_history` 等；`sanitizeInput` 的提示注入正则只覆盖少量英文句式，易绕过却制造「已防护」错觉。

**修复方向**：

- **把真正的边界交给 F1 的隔离方案**，不要依赖字符串匹配当主防线。
- 文件访问改为**白名单式**（限定目录 + 允许的扩展名），而非黑名单。
- 提示注入检测降级为「监测 / 告警 / 记录」用途，而不是唯一拦截手段，避免误判合法输入又给出安全错觉。

**验收标准**：即便绕过字符串检测，用户也无法读到隔离边界外的文件（依赖 F1 成立）。

---

### F7 — 安全响应头与错误信息收敛（中）

**问题本质**：未使用 `helmet` 等安全头中间件；大量路由把 `(err as Error).message` 原样返回给客户端，可能泄漏内部路径与实现细节。

**修复方向**：

- 引入 `helmet`，配置合理的 CSP、`X-Content-Type-Options`、`Referrer-Policy` 等。
- 对外统一返回通用错误信息（如「请求处理失败」），详细堆栈只写服务端日志并附请求 ID 便于排查。

---

### F8 — 补齐根级 `.gitignore`，核查历史泄漏（低）

**问题本质**：项目根目录没有 `.gitignore`，`data/users.json`、`data/admin-key.txt`、会话 `.jsonl` 等敏感产物未被忽略（实际 git 根在上层 `/Users/kumo/git`）。

**修复方向**：

- 添加根级 `.gitignore`，忽略 `data/`、`workspaces/`、`*.env`、`*.log`、会话文件、`node_modules`、`dist` 等。
- 核查 git 历史是否曾提交过上述敏感文件；若有，需清理历史并轮换泄漏的密钥。

---

### F9 — 会话超时读取配置并加绝对上限（低）

**问题本质**：`SESSION_TIMEOUT_MS` 硬编码 24 小时，未读取 `config.sessionTimeoutHours`；会话只有空闲滑动过期、无绝对过期上限。

**修复方向**：

- 让超时时长读取 `config.sessionTimeoutHours`，允许按环境收紧。
- 除滑动过期外，增加**绝对过期时间**（会话创建后固定时长必须重新登录），缩小令牌泄漏后的利用窗口。
- 保留 / 完善用户维度的会话吊销能力（改密钥、角色变更时清理会话，现已部分实现）。

---

## 建议的执行顺序

1. 先做 F2（清理并轮换已泄漏的明文主密钥）——这是当前就已经暴露的机密。
2. 再做 F1（隔离），因为它同时决定了 F6 是否还有意义。
3. 然后 F3 / F4 / F5 收敛外部攻击面。
4. 最后 F7 / F8 / F9 做常规加固。

每完成一项，建议按各节的「验收标准」做一次针对性验证，再进入下一项。
